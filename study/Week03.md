# 🚀 [Week 3] API, REST 아키텍처 및 Spring 핵심 개념 정리

## 1. API와 REST 아키텍처

### 📌 API (Application Programming Interface)
소프트웨어 간 데이터를 주고받기 위한 통신 규칙을 정의한 인터페이스입니다.
* **역할:** 클라이언트와 서버 사이에서 요청과 응답을 연결하는 매개체
* **예시:** 날씨 앱(클라이언트)에서 기상청 서버(서버)에 날씨 정보를 요청하면, 이를 처리하여 응답으로 돌려주는데 이때 소통을 위한 규칙이자 도구가 바로 API입니다.

### 📌 REST (Representational State Transfer)
API를 설계하는 방식으로, **자원을 URI로 표현**하고 **HTTP Method로 자원을 조작**하는 HTTP 기반 설계 방식입니다.
* **URI:** URL(위치 기반 식별) + URN(이름 기반 식별)으로 자원을 식별하기 위한 논리적, 물리적 주소
* **자원(Resource):** 관련된 데이터와 그 의미를 하나로 묶은 객체와 유사한 개념
* **동작 원리:** 사용자가 특정 자원을 조회하면, 서버는 그 순간의 자원 상태를 스냅샷을 찍어 JSON 형태로 전달합니다. 특정 주소에 묶여있는 데이터를 표준화된 행위로 조작하는 것입니다.

### 📌 REST의 3가지 구성 요소
1. **자원 (Resource):** 데이터 (URI로 표현)
2. **행위 (Verb):** 데이터를 조작하는 법 (HTTP Method)
3. **표현 (Representations):** 데이터의 구조 (JSON, XML 등)

### 📌 REST의 핵심 특징
* **Client-Server 구조:** 클라이언트는 화면, 서버는 데이터 및 비즈니스 로직을 담당하여 역할을 분리합니다.
* **무상태성 (Stateless):** 서버는 클라이언트의 이전 상태를 기억하지 않습니다. (예: JWT를 통한 인증)
* **인터페이스 일관성 (Uniform Interface):** 조작 구조가 일관되어야 합니다.
* **캐시 가능 (Cacheable):** 서버의 응답 데이터를 클라이언트나 중간 서버가 기억해두고 재사용할 수 있습니다.
* **계층 구조 (Layered System):** 클라이언트와 서버 사이 여러 중간 계층이 존재할 수 있으며, 클라이언트는 서버의 내부 구조를 알 필요가 없습니다.
* **자체 표현성 (Self-descriptiveness):** 데이터 메시지만 보고도 무엇을 어떻게 처리해야 하는지 알 수 있어야 합니다.

## 2. HTTP Method와 CRUD 설계

REST는 URI와 HTTP Method를 사용하여 자원의 CRUD를 설계합니다.

* **Create (POST):** 자원 생성 (회원가입, 로그인 요청, 게시글/댓글 작성)
* **Read (GET):** 자원 조회 (사용자/게시글/댓글 조회)
* **Update (PATCH):** 부분 수정 (비밀번호, 닉네임, 게시글 일부 내용 변경)
* **Update (PUT):** 전체 수정 (게시글 등 자원을 통째로 교체할 때 사용)
* **Delete (DELETE):** 자원 삭제 (회원 탈퇴, 게시글/댓글 삭제)

### 💡 핵심 비교 포인트
* **PUT vs PATCH:** `PUT`은 멱등성(Idempotency, 여러 번 수행해도 결과가 같음)을 가지지만 `PATCH`는 구현에 따라 다를 수 있습니다. 이를 통해 재시도 가능 여부를 판단합니다.
* **논리적 삭제(Soft) vs 물리적 삭제(Hard):**
    * Soft Delete: DB에서 지우지 않고 상태만 변경하여 사용자 접근을 막음 (복구 가능)
    * Hard Delete: DB에서 데이터를 완전히 삭제함 (복구 불가)

## 3. RESTful API 설계 및 문서화

### 📌 RESTful API 설계 원칙
* 동사 대신 **명사** 사용, 단수보다는 **복수형** 사용
* 슬래시(`/`)로 **계층 관계** 표현 (마지막에는 슬래시 포함 X)
* URI는 **소문자**로 작성하며, 파일 **확장자(.png, .jsp 등)는 포함하지 않음**
* 행위가 아닌 자원 중심의 설계, 역할 분리 및 확장성 고려

### 📌 API 명세서 (API Specification)
동작 방식, Endpoint, 요청/응답 구조, 인증 방식 등을 정리해 API 사용 방식을 정의한 문서입니다.
* **목적:** 명확한 소통, 일관성 유지, 효율적인 유지보수
* **구조:** 기능, HTTP Method, EndPoint, 버전, 개발 여부, 파라미터, 응답 구조, 오류 대응 방법 등
* **인증 방식:**
    * `Basic Auth`: 아이디/비밀번호 방식
    * `Bearer Token`: 토큰(JWT 등) 기반 인증
    * `API Key`: 고유 키 발급 인증
    * `OAuth`: 외부 서비스 기반 인증 (소셜 로그인 등)

## 4. Spring 핵심: Bean과 DI

### 📌 Spring Bean과 컨테이너
자바와 달리 Spring은 IoC 컨테이너에 객체(Bean)를 미리 만들어놓고 **싱글톤(Singleton)**으로 관리합니다.

* **수동 등록:** 외부 라이브러리 객체를 등록할 때 `@Configuration` 클래스 안에서 `@Bean` 사용
* **자동 등록:** `@Component` 등 컴포넌트 스캔 대상 어노테이션이 붙은 클래스

### 📌 DI (의존성 주입, Dependency Injection)
필요한 객체를 직접 생성(`new`)하지 않고 **Spring 컨테이너가 직접 주입**해주는 설계 방식입니다.
* *예시:* 컨트롤러가 실행될 때마다 서비스 객체를 생성하는 것이 아니라, 스프링이 미리 만들어 잘 관리하고 있는 객체 하나를 주입받아 사용합니다. (결합도 감소)

#### DI 주입 방식
1. **생성자 주입 (권장 ⭐):** 불변성을 보장하여 서비스 객체가 실행 중 바뀔 위험이 없습니다.
2. **필드 주입:** 객체를 외부에서 주입할 수 없어 스프링 프레임워크 밖(테스트 등)에서 사용이 제한적입니다.
3. **Setter 주입:** 실행 중에 의존성을 변경해야 할 특수한 경우에만 사용합니다.

## 5. 어노테이션(Annotation)과 Lombok

어노테이션은 코드에 특별한 의미와 기능(메타데이터)을 부여하는 기술입니다.

### 📌 Spring 핵심 어노테이션
* `@Component`: 스프링이 스캔하여 자동으로 Bean으로 등록
* `@Configuration`: 설정 클래스임을 명시
* `@Bean`: 개발자가 직접 제어 불가능한 외부 라이브러리 등을 수동으로 Bean 등록할 때 사용
* `@Transactional`: 메서드 내의 로직을 하나의 트랜잭션으로 묶어 원자성 보장
* `@Value`: `application.yml` 등의 외부 설정 값을 변수에 주입

### 📌 Lombok 라이브러리
반복되는 코드(Getter, Setter, 생성자 등)를 컴파일 시점에 자동으로 생성해줍니다.
* `@Getter` / `@Setter`: 필드 값의 조회 및 수정 메서드 자동 생성
* `@Builder`: 매개변수가 많은 객체 생성 시 가독성을 높여주는 빌더 패턴 자동 적용
* `@NoArgsConstructor`: 기본 생성자 자동 생성
* `@AllArgsConstructor`: 모든 필드를 매개변수로 받는 생성자 자동 생성
* `@RequiredArgsConstructor`: `final`이 붙은 필수 필드에 대한 생성자 자동 생성 (DI에 주로 사용)

## 6. Database: 트랜잭션의 특징 (ACID)
데이터베이스의 상태를 변화시키는 하나의 논리적 기능을 수행하기 위한 작업 단위입니다.

* **A (Atomicity, 원자성):** 트랜잭션 내의 연산은 모두 성공하거나 모두 실패해야 합니다. (All or Nothing)
* **C (Consistency, 일관성):** 트랜잭션이 성공적으로 완료되면 데이터는 항상 정상적이고 일관된 상태를 유지해야 합니다.
* **I (Isolation, 격리성):** 둘 이상의 트랜잭션이 동시에 실행될 때, 서로 독립적이며 영향을 주지 않아야 합니다.
* **D (Durability, 지속성):** 성공적으로 수행된 트랜잭션의 결과는 영구적으로 데이터베이스에 반영 및 저장되어야 합니다.

## 7. Swagger API 문서화 도구

### 📌 SwaggerConfig
API 명세서를 어떤 식으로 만들고 어디까지 공개할 것인지 세부 지침을 전달하는 **설정 클래스**입니다.

1. **설정 요소:**
    * 메타데이터 정의: 문서 제목, 설명, 버전 등
    * 보안 및 인증: JWT, API Key 등의 보안 스키마 설정
    * 스캔 범위 및 필터링: 패키지 경로를 지정하여 문서화할 컨트롤러 필터링
    * 서버 환경 설정: 로컬, 개발, 운영 서버 주소 명시
2. **동작 흐름:**
    * 설정 파일에서 `OpenAPI` 객체 수동 생성
    * 스프링 구동 시 소스 코드 스캔 (컨트롤러, 매개변수, 리턴 타입 확인)
    * 스캔한 정보를 바탕으로 Swagger-UI가 화면 렌더링
    * `/swagger-ui/index.html` 등의 특정 경로로 자동 서빙
