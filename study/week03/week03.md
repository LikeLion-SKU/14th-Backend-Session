## API와 REST API

* **API (Application Programming Interface):** 소프트웨어 간 데이터를 주고받기 위한 통신 규칙을 정의한 인터페이스
* **REST (Representational State Transfer):** 자원을 URI로 표현하고, HTTP Method로 자원을 조작하는 HTTP 기반 설계 방식
  > **URI (Uniform Resource Identifier):** 서버에 있는 자원을 식별하는 고유한 주소

### REST의 구성
1. **자원 (Resource):** 서버가 관리하는 데이터(사용자, 게시글 등)를 의미하며, URI로 표현 및 구분된다. 클라이언트는 이 URI를 통해 자원을 요청한다.
2. **행위 (HTTP Method):** 자원에 대해 어떤 동작을 할지 표현하는 수단이다. 주요 Method로 GET, POST, PUT, PATCH, DELETE가 사용된다.
3. **표현 (Representation):** 클라이언트와 서버가 데이터를 주고받는 형태를 말하며, 주로 JSON이나 XML 형식을 사용한다.

### REST의 주요 특징
* **Client - Server 구조 (클라이언트 서버 구조):** 클라이언트(화면, 사용자 정보 관리)와 서버(데이터 및 비즈니스 로직 담당)의 역할이 명확히 분리된다.
* **Stateless (무상태):** 서버는 이전 요청을 기억하지 않음. 매 요청마다 필요한 정보 포함.
* **Uniform Interface (인터페이스 일관성):** 자원은 URI로, 행위는 HTTP Method로 표현하여 구조가 일관됨.
* **REST API:** REST의 설계 원칙을 따르는 API
  > **RESTful API:** REST의 설계 원칙을 "잘 지킨" API.

## CRUD와 HTTP Method
* **CRUD:** 데이터 관리 및 조작에서 수행되는 4가지 기본 작업
* **HTTP Method:** 클라이언트가 서버에게 "어떤 동작을 할 것인지" 요청하는 방식
- **C**reate (생성) -> **POST**
- **R**ead (조회) -> **GET**
- **U**pdate (수정)
    - **PUT:** 데이터 전체 수정 (항상 멱등성을 보장)
    - **PATCH:** 데이터 부분 수정 (요청 방식에 따라 멱등성이 보장되지 않을 수 있음)
- **D**elete (삭제) -> **DELETE**
    * *Soft Delete:* 논리적 삭제 (boolean값으로 데이터 사용여부를 결정, 복구 가능)
    * *Hard Delete:* 물리적 삭제 (데이터베이스에서 완전히 삭제, 저장 공간 확보)
### REST와 CRUD, HTTP Method의 관계
* REST는 URI로 자원을 표현하고, HTTP Method로 자원의 동작을 정의한다.
* 이때 HTTP Method는 CRUD와 대응된다.
* **→ 즉, REST는 URI와 HTTP Method를 통해 CRUD를 수행하는 설계 방식이다**.

## RESTful URI 설계 원칙
1. **명사**을 사용한다.
2. **복수형**을 사용한다.
3. 계층 구조를 표현한다.
4. 마지막에 슬래시(`/`)를 사용하지 않는다.
5. **소문자**를 사용한다.
6. 파일 확장자(`.jpg` 등)를 포함하지 않는다.

### 왜 REST는 관계 중심 URI를 사용할까?
1. **REST는 자원 중심 설계:** URI는 "무엇을 할까"가 아니라 "어떤 자원인가"를 표현
2. **URI는 자원의 식별자:** URI는 데이터를 찾기 위한 주소
3. **확장성과 재사용성:** 같은 패턴으로 확장이 가능
4. **URI와 HTTP Method 역할 분리:** 하나의 URI로 다양한 동작 처리가 가능

### API 명세서란?
* API의 동작 방식, Endpoint, 요청 및 응답 구조, 인증 방식 등을 정리하여 API 사용 방법을 명확하게 설명하는 문서
* **왜 필요할까?**
  * **명확한 소통:** 개발자 간의 소통이 원활해지고, 해당 API를 사용하는 외부 개발자도 쉽게 이해할 수 있다.
  * **일관성 유지:** 명세서를 통해 API 설계와 구현에서 일관성을 유지할 수 있다.
  * **효율적인 유지보수:** 잘 작성된 명세서는 코드 수정 시 변경 사항과 영향 범위를 쉽게 파악할 수 있다.

### API 문서화 도구
* **API 문서화 도구:** Swagger, Postman, GitBook, Spring Rest Docs
* **Swagger:** REST API를 설계, 문서화하고 테스트까지 할 수 있는 프레임워크
  * 코드만 작성하면 문서가 자동으로 생성, API 변경 시 문서도 자동 변경, 브라우저에서 직접 테스트가 가능
  
## Spring의 객체 관리 방식
* **Bean:** 개발자가 직접 `new`로 만들지 않고, Spring Container가 대신 생성하고 관리해 주는 객체 (기본적으로 싱글톤 관리), 동일 객체 재사용 가능.
> 효과: 객체를 한 곳에서 중앙 관리, 코드의 변경 사항을 쉽게 반영 가능, 유지보수 쉬워짐.
* **DI (의존성 주입, Dependency Injection):** 필요한 객체를 직접 생성하지 않고 Spring이 대신 주입해 주는 방식

### Bean 등록 방법
1. **자동 등록:** `@Component`, `@Service`, `@Controller` 등의 어노테이션이 붙은 클래스는 Spring이 자동으로 Bean으로 등록 (코드가 간결함)
2. **수동 등록:** 개발자가 직접 Bean을 등록하는 방식 (외부 라이브러리 객체를 등록할 때 사용)

### DI (의존성 주입) 3가지 방식
1. **생성자 주입 [권장]:** 생성자를 통해 필요한 객체를 전달받는 의존성 주입 방식. `final` 키워드 사용이 가능하여 불변성을 보장하고, 생성 시 의존성이 확정되어 안정성이 높음.
2. **필드 주입:** 필드에 직접 `@Autowired`를 사용하여 의존성을 주입하는 방식.
3. **Setter 주입:** Setter 메서드를 사용하여 의존 객체를 주입받는 방식. 선택적 의존성이나 변경 가능성이 있을 때 사용.

## Annotation(@)
* **Annotation:** Java에서 코드에 특별한 의미와 동작을 부여하는 기술, 코드에 대한 정보와 설정을 제공하는 메타데이터.
### 기본 Lombok Annotation
> Lombok: 반복되는 코드를 자동으로 생성해주는 라이브러리
* `@Getter`: 클래스 필드에 대한 getter 메서드를 자동으로 생성해 준다. 필드 값을 조회할 때 사용한다.
* `@Setter`: 클래스 필드에 대한 setter 메서드를 자동으로 생성해 준다. 필드 값을 수정할 때 사용한다.
* `@Builder`: 객체 생성 시 값을 하나씩 설정할 수 있는 방식으로, 많은 매개변수를 가진 객체를 생성할 때 유용하다.
* `@NoArgsConstructor`: 파라미터가 없는 기본 생성자를 자동으로 생성해 준다.
* `@AllArgsConstructor`: 클래스에 선언된 모든 필드를 매개변수로 받는 생성자를 자동으로 생성해 준다.
* `@RequiredArgsConstructor`: final 필드에 대한 생성자를 자동 생성하고, 필수 의존성이 반드시 초기화하도록 강제한다.
* `@Transactional`: 여러 DB 작업을 하나로 묶어 모두 성공하거나 모두 실패하도록 처리하는 기능
### Transaction의 4가지 특성 (ACID)
* **A (Atomicity, 원자성):** 트랜잭션 내의 모든 작업은 모두 성공하거나 모두 실패해야 한다.
* **C (Consistency, 일관성):** 트랜잭션 수행 전후에 데이터는 항상 정상적인 상태를 유지해야 한다.
* **I (Isolation, 격리성):** 트랜잭션은 서로 독립적으로 실행되어야 하며, 동시에 실행되는 다른 트랜잭션의 영향을 받지 않아야 한다.
* **D (Durability, 지속성):** 트랜잭션이 완료되면 그 결과는 영구적으로 저장되어야 한다.
