>## API를 어떤 기준으로 어떻게 설계해야할까? -> *REST*

>## REST(Represntational Sate Transfer)란?  
>### 자원을 URI로 표현하고, HTTP Method로 자원을 조작하는 HTTP 기반 설계 방식   
>> 모든 데이터는 자원  
>> URI : 서버에 있는 자원을 식별하는 고유한 주소  

>## REST의 구성

>>### 1. 자원 -> URI로 표현
>>### 2. 행위
>>### 3. 표현

>## PUT VS PATCH
>>### Resource를 전체 교체 VS Resource를 일부 수정   
>>### 항상 멱등성 보장 VS 멱등성 항상 보장 X

>## HTTP 메소드에서 멱등성이 필요한 이유
>>### 요청 실패 시 재시도 가능 여부 판단 기준
>>### 멱등하지 않은 경우, 중복 요청으로 문제 발생 가능
>>> 너무 복잡하다면?   
>>> PATCH보다는 PUT으로 전부를 수정하는 것이 더 적절 할 수 있음   
>>> (PATCH는 정확히 기존 값을 알아야하기에, PUT이 더 효율적일 수 있음)

>## Soft(논리) Delete vs Hard(물리) Delete
>> 데이터베이스에서 데이터 삭제 X, 사용자가 접근 불가   
>> VS   
>> 데이터베이스에서 실제로 데이터 삭제 O

>## REST 설계 원칙(RESTful URI 설계 원칙)
>>### 1. 명사를 사용한다 (/create-user X, /posts/1 O)
>>### 2. 복수형을 사용한다 ( /post/1 X, /posts/1 O)
>>### 3. 계층 구조를 표현한다(/users/3/posts O)
>>### 4. 마지막 슬래시를 사용하지 않는다(/posts/ X, /posts O)
>>### 5. 소문자를 사용한다(/Users X, /users O)
>>### 6. 확장자를 포함하지 않는다(/photos.jpg X /photos O)

>## 왜 REST는 관계 중심 URI를 사용할까?
>## *확장성과 역할 분리 관점*
>>### 1. REST는 자원 중심 설계(무엇을 할까 X, 어떤 자원인가 O)
>>### 2. URI는 자원의 식별자(URI는 데이터를 찾기 위한 주소)
>>### 3. 확장성과 재사용성   
>> (행위 중심으로 하면, 같은 패턴으로 확장이 불가능 하고, API 계속 새로 만들어야함)
>>### 4. URI와 HTTP Method 역할 분리
>> (하나의 URI로도 다양한 동작 수행 가능, URI에 get, post등 포함 X)

>## DI(의존성 주입)
>>### 필요한 객체를 직접 생성하지 않고, Spring이 대신 주입
>>## 사용 이유
>>### 1. 객체간 결합도 감소
>>### 2. 코드 테스트 용이
>>### 3. 유지보수 편리
>>### 4. Spring이 객체 관계를 자동으로 연결
>
>>## DI 방식
>>## 1. 생성자 주입 → 주로 사용
>>### 생성자를 통해, 필요한 객체를 전달 받음
>>### final 키워드 사용 가능 → 불변성 보장
>>### 객체 생성 시 의존성 확정
>>## 2. 필드 주입
>>### @Autowired를 생성자가 아닌 필드에 직접 사용
>>## 3. 세터 주입
>>### 세터 메소드에 @Autowired를 사용
>>### → 선택적 의존성 OR 변경 가능성 존재 시 사용

>## Lombok
>### 반복되는 코드를 자동으로 생성해주는 라이브러리
>>## 기본 Lombok Annotation
>>### @Getter → 게터 자동생성
>> Getter : 필드 값을 조회할 때 사용
>>### @Setter → 세터 자동생성
>> Setter : 필드 값을 수정할 때 사용
>>### @Builder → 객체 생성 시 값을 하나씩 설정 가능
>
>>## 생성자 관련 Lombok Annotation
>>### @NoArgsConstructor → 파라미터가 없는 기본 생성자 자동 생성
>>### @AllArgsConstructor → 클래스에 선언 된 모든 필드를 매개변수로 받는 생성자를 자동 생성
>>### @RequiredArgsConstructor → final 필드에 대한 생성자를 자동 생성(필수 의존성이 반드시 초기화 되도록 강제)
>>#### → 객체 생성 시점에 필요한 의존성이 모두 주입 되지 않으면, 컴파일 단계 오류 발생
>>#### → RequiredArgsConstructor를 이용해 컴파일(런타임X) 문제를 발견 가능

>## @Transactional
>### 여러 DB 작업을 하나로 묶어 모두 성공하거나, 모두 실패하도록 하는 기능
>## 동작 방식
>### 메소드를 하나의 작업 단위로 실행
>### 정상 실행 → Commit(DB에 정상적으로 반영)
>### 중간에 오류 발생 → Rollback(이전 상태로 복구)
>### → 이를 통해 일부만 작업 되는 상황 방지
>## Transactional의 필요성
>### 하나의 메소드 안에서 여러 DB 작업이 함께 실행 될 때,
>### → 하나라도 실패하면 전체 작업을 취소 해, 데이터의 일관성을 유지
>>### ex) 은행 송금과 같은 상황에서 적용
>>#### 한 쪽 계좌에서 출금이 되었으나, 입금 받는 쪽 계좌에서 오류 발생 시
>>#### → 데이터 일관성 붕괴
>## Transactional의 특성 → ACID
>### A(원자성) → 트랜잭션은 원자적으로 처리 되어야 함(모두 성공 OR 모두 실패)
>### C(일관성) → 트랜잭션 수행 전 후, 데이터는 항상 정상적인 상태 유지
>### I(격리성) → 트랜잭션은 서로 독립적으로 실행 + 동시에 실행되는 다른 트랜잭션의 영향 X
>### D(지속성) → 트랜잭션이 완료 되면, 그 결과는 영구적으로 저장되어야 함
